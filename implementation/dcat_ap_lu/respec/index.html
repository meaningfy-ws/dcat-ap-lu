DCAT-AP-LU is a Luxembourg-specific extension of the DCAT-AP profile used for sharing information about Catalogues containing Datasets and Data Services descriptions in Europe. It was created as part of the national implementation of the Data Governance Act (DGA) to support the creation of a single national dataset catalogue for all public sector data. The DCAT-AP-LU reuses large parts of the DCAT-AP-v3.0.1 standard, with a set of minor adaptations to support specific requirements for Luxembourg.

Introduction

Context

DCAT-AP-LU is a Luxembourg-specific extension of the [[DCAT-AP-v3.0]] profile used for sharing information about Catalogues containing Datasets descriptions in Europe. It was created as part of the national implementation of the Data Governance Act (DGA) to support the creation of a single national dataset catalogue for all public sector data. The DCAT-AP-LU reuses large parts of the [[DCAT-AP-v3.0]] standard, with a set of minor adaptations to support specific requirements for Luxembourg.

Scope of the Application Profile

This document provides guidelines for how to use DCAT-AP-LU to describe datasets in the context of the DGA national implementation but that are also interoperable with other national and international dataset catalogues following the [[DCAT-AP-v3.0]] specification.

Revision history

The first DCAT-AP-LU specification was released in March 2025. It focuses exclusively on the Dataset and Distribution concepts, as well as the most relevant supportive entities where these are reused "as is" from [[DCAT-AP-v3.0]] or where their specification differs for DCAT-AP-LU. A more comprehensive version, including additional entities such as Catalogue, Data Service, Dataset Series and further main and supportive entities will be released by Q1 2026.

Status

This application profile is published by a collaboration of the Luxembourg National Data Service (LNDS) and the Ministère de la Digitalisation, Luxembourg. It is being developed with input and feedback from relevant national stakeholders.

It has no official standing of any kind and does not represent the support or consensus of any standards organisation.

Conformance Statement

In order for applications to conform to DCAT-AP-LU, it MUST conform to [[ DCAT-AP-v3.0]]. In addition, the application must conform to the mentioned constraints and usage guidelines following similar conformance statements as specified in [[DCAT-AP-v3.0]].

Terminology

An Application Profile is a specification that reuses terms from one or more base standards, and adds more specificity by identifying mandatory, recommended and optional elements for a particular application, as well as advising on the use of controlled vocabularies.

A Dataset is a collection of data featured in a data catalogue that describes datasets and offers services to facilitate their discovery and reuse.

A Distribution is a dataset that is available in a specific format. If a dataset is published in multiple formats (e.g. Excel, CSV, Data Cube), then they are described as separate distributions.

Overview

DCAT-AP-LU is a DCAT application profile for sharing information about Catalogues containing Datasets descriptions in Luxembourg, compatible with the [[DCAT-AP-v3.0]] profile used in Europe. The core classes of the Application Profile are thus the classes Dataset and Distribution, with Catalogue and Data Service to be added in the next iteration.

This document describes the usage of the following main entities for a correct usage of the vocabulary:

General usage guidelines

The DCAT-AP-LU profile was developed in the context of the national DGA implementation but is intended to be usable in other contexts. The usage guidance is subdivided accordingly.

Usage guide on Datasets and Distributions

The general usage guide for [[DCAT-AP-v3.0]] applies for most of DCAT-AP-LU. The main differences for this profile include:
  • Dataset.identifier: This property is mandatory to promote interoperability and data governance best practice. For the Luxembourg public sector, an identifier strategy is under development.
  • Dataset.dataProtectionType: This new property is applicable in the DGA context only (see below).
  • Dataset.inSeries: This property was upgraded from optional to recommended. More guidance to follow.
  • Dataset.rights: The accepted values for this property are restricted to "public" and "non-public".
  • Dataset.type: The accepted values for this property have been adapted as described in [[[#controlled-vocs]]].
  • Distribution.mediaType: This property was upgraded from optional to recommended due to the greater granularity of the associated vocabulary compared to Distribution.mediaType.
  • Distribution.hasPolicy: DCAT-AP-LU provides extended usage guidance for expressing data use restrictions using [[ODRL]] and [[DUO]].
Except for the properties in the list above, the property type (mandatory, recommended or optional) was adopted as-is from [[DCAT-AP-v3.0]]. The overall format of this specification only expressly highlights whether a property is mandatory or not, via the `Card` column in the Main Entities tables. The further distinction between recommended and optional properties, while widely used in DCAT-AP, is not considered. Please refer to the Quick reference table in Appendix A in [[DCAT-AP-v3.0]] for an overview of recommended and optional properties.

Usage guide for the DGA context

Dataset.dataProtectionType

Any metadata record for a protected dataset published in the context of the DGA must include at least one of the 6 controlled vocabulary terms listed in [[[#controlled-vocs]]] (:PersonalData, :CommerciallyConfidentialData, :IntellectualPropertyData, :StatisticallyConfidentialData, :SpecialCategoryPersonalData, :ProfessionalConfidentialData). These properties originate from, and are maintained by, the Data Privacy Vocabulary [[DPV]].

Dataset.applicableLegislation

Any metadata record for a dataset published in the context of the DGA must include a reference to the relevant law articles (links TBA) to define whether the dataset may be reused under the definition of "TITRE V – Traitement ultérieur de données à caractère personnel par les entités publiques" (TUDP), "TITRE VI –Accès et réutilisation des données détenues par des organismes du secteur public par des réutilisateurs dedonnées" (ARD) or both. A metadata record that does not include at least one of these references will notbe considered valid. Exact [[ELI]] compliant URIs for the different contexts will be included here once the lawtext has been officially adopted and published.

Dataset.sample

In order to facilitate the creation and assessment of data access requests for datasets in the DGA context, the associated metadata record should include data dictionaries whenever possible. Data dictionaries should be included via a special type of distribution referenced from the Dataset.sample property. This guidance reuses the implementation model proposed by the HealthDCAT-AP metadata profile.
No specific format is imposed for the data dictionary. Machine-readable tabular formats such as CSV or TSV are preferred. A minimal set of properties to be included in a data dictionary is under development and will be made available in due course.

Distribution.hasPolicy

DCAT-AP-LU leverages existing mechanisms from [[ODRL]] and [[DUO]] to enable data holders to specify additional restrictions on data reuse in a consistent and machine-readable fashion. The following points are expressed:
  • Restrictions can be expressed as either permissions or prohibitions.
  • The default odrl:policy type is odrl:Set but policies of type odrl:Offer may also be used.
  • The default odrl:action for all scenarios is odrl:use
  • Permitted constraint types (odrl:leftOperand) for constraints are odrl:Purpose/dpv:Purpose, dateTime, spatial and industry.
  • Permitted constraint modifiers (odrl:rightOperand) are:
    • Temporal restrictions: an [[ISO8601]] compliant Date and Time value.
    • Spatial restrictions: a valid geospatial area expression such as an [[ISO3166]] Country Code.
    • Purpose restrictions: one of the data use restrictions specified in the table below.
    • Industry restrictions: list under development
